# SportsAI — Architecture & Data Strategy

## How to Make This Useful to a Common User — The Improvement Loop

### The core insight
Users don't improve from numbers ("your arm angle is 87°").
They improve from three things, in this order:
1. SEEING themselves vs correct movement (visual comparison)
2. Knowing what it should FEEL like (body cue, not a number)
3. A specific drill that isolates the exact failure pattern

Use BOTH the 3D skeleton overlay AND verbal cues. The skeleton shows WHAT is wrong
spatially. The feel cue tells them HOW to fix it with a mental model they can use
in real time during practice. Numbers alone are useless to a common user.

### The feedback loop that actually produces improvement
Session → annotated video → feel cue → drill → next session → compare

1. User uploads video
2. /analyse/video returns an annotated MP4 they can watch immediately:
   - Green joints = correct, Red joints = problem
   - Grey ghost skeleton = where elite athletes are at that same moment
   - Key moment highlighted with the feel cue as text overlay
   - HUD panel showing all metrics + next session focus
3. Claude generates: headline + strengths + max 3 fixes with:
   - What is actually happening in their body (1 sentence)
   - What it should FEEL like (sensation language, not angle numbers)
   - Exact drill with sets/reps
   - Why the drill works biomechanically
4. If same metric is bad for 3 sessions → drill prescription escalates
5. Progress tracker shows trend (improving/plateau/declining) per metric
6. Injury risk flags trigger before they get hurt

### Why this works better than Balltime
- Balltime shows you what happened. This shows you what to DO about it.
- Feel cues work in real-time during practice — you can't check an angle mid-jump
- Drill library is technique-specific, not generic
- Persistent problem detection means the system notices when you're stuck
- Ghost overlay = direct visual comparison, not abstract numbers

### Skill level adaptation
- Beginner: everyday language, simple cues ("pull your elbow back like drawing a bow")
- Intermediate: coaching language, phase references ("load your hips on the penultimate step")
- Advanced: biomechanical language ("increase shoulder abduction angle at arm cocking phase")

## Why Volleyball First
- Court is flat, camera angles are mostly fixed (sideline or endzone)
- Actions are discrete and well-defined: serve → pass → set → attack → block → dig
- This makes the CV pipeline dramatically simpler to train and validate vs. basketball or soccer

Balltime AI (acquired by Hudl) proved the market — 67K+ users, App Store top charts. Their weaknesses:
- Misidentifies the correct player ~50% of the time
- Aggressive watermarks
- Poor customer support

Our opening: better accuracy + better UX + team video focus they don't do well.

"Volleyball first, then expand" is textbook product discipline. One rules engine that works extremely
well for one sport → revenue → fund the next sport. Launching 5 sports at once guarantees mediocrity.

## System Architecture (5 Layers)

### Layer 1 — Shared Pose Backbone (train ONCE, use for all sports)
- MMPose (OpenMMLab) — multi-person, 133-keypoint whole-body, 3D mesh recovery, Apache 2.0
- AlphaPose — 70+ mAP on COCO, 60+ mAP on PoseTrack, efficient online multi-person tracker
- MediaPipe — 33 keypoints, 30fps, on-device, zero training needed — plug in directly
- Pre-trained on 200M+ human poses
- Status: Currently using YOLO11x-pose → upgrade path is MMPose + AlphaPose for multi-person

### Layer 2 — Biomechanics Engine (fine-tune per sport category)
- Joint angles, velocity, acceleration, symmetry, balance — computed from keypoints
- ~3 categories: individual precision, team ball, combat
- ~500–2K labeled clips each
- Status: Built for spike (arm_cock_angle, jump_height, approach_speed, contact_point, follow_through)
  Serve/block/dig biomechanics extraction implemented, thresholds pending data collection

### Layer 3 — Sport-Specific Rules Engine (one per sport)
- What does "correct form" mean for THIS sport?
- Written by coaches + sports scientists
- Only this layer is unique per sport — everything else is shared infrastructure
- Status: Spike analyser built. Unified analyser.py covers all 4 techniques.
  Team video needs: jersey re-ID, formation recognition (P1–P6 rotations), rally segmentation

### Layer 4 — AI Coach Feedback (Claude API)
- Takes biomechanics scores + athlete history → generates personalized coaching
- Example: "Your left knee collapsed 12° on the third rep — classic fatigue pattern. Here's the fix..."
- Status: Not built yet

### Layer 5 — Progress Loop
- Track specific metrics over time, score each session
- Compare to elite reference models
- Prescribe drills, detect plateaus, flag injury risk patterns
- Adapt to the athlete
- Status: Not built yet

## Training Strategy — Fastest Path to Olympic-Grade

Do NOT train from scratch (12–18 months). Use this path instead:

### Week 1–2 — Plug in pre-trained backbones
- MMPose as skeleton backbone (multi-person, state-of-the-art, Apache 2.0)
- AlphaPose for real-time multi-person tracking layer
- Both work out of the box — zero labeling needed at this stage

### Week 2–4 — Fine-tune on volleyball-specific data
- VolleyVision: 14K-image volleyball actions dataset, YOLOv8m, 7 classes
  (defense-move, attack, block, reception, service, setting, stand)
- volleyball_analytics repo: end-to-end FastAPI pipeline, VideoMAE game state,
  6-action detection (serve, receive, set, spike, block, ball), court segmentation
- Roboflow Universe: volleyball actions dataset with code, models, metrics

### Week 4–8 — Team video additions
These are NOT covered by the above repos and must be built:
- Jersey number + player re-identification across camera cuts (hardest part)
  → Use PaddleOCR (free, fast) for jersey OCR
- Formation recognition — detecting rotation positions (P1–P6)
- Ball trajectory tracking with speed estimation
  → Use TrackNet (open source, designed for fast-moving small balls in sports)
- Rally segmentation — detecting when ball is in/out of play

### Full stack
- YOLOv8-pose → fine-tune on Roboflow volleyball dataset
- DeepSORT or ByteTrack → multi-player tracking (need 12-person: 6v6)
- PaddleOCR → jersey number recognition
- TrackNet → ball trajectory
- Claude API → intelligent feedback generation (Layer 4)

Realistic timeline:
- Working demo: 6–8 weeks using open-source repos above
- Olympic-grade accuracy (rival broadcast analytics tools): 4–6 months iterative fine-tuning

## Architecture Gap: 1v1 vs Team Video

Current system is designed for individual technique analysis (1 athlete per video).
Team video is a different architecture:
- Replace 2-person tracker with 12-person tracker (6v6)
- Add volleyball scoring logic (rotations, faults, libero rules)
- Add rally segmentation and game-state detection (SERVICE / PLAY / NO-PLAY)
- Formation recognition per rotation position

## Generate Your Own Data — The Flywheel

| Method | Sports | Details | Scale |
|--------|--------|---------|-------|
| Your own users | All | Every upload = training data (ToS consent). 1,000 users = ~3,000 labeled clips/week. "Was this feedback helpful?" = label signal. Compounds forever | Grows forever |
| Coach labeling program | All | 5–10 certified coaches per sport at $20–30/hr, 500 clips each. Cost: ~$5,000/sport. Exact approach Hudl and CoachAI used early on | 500–2K/sport |
| Scale AI / Label Studio | Any | Crowdsource annotation. Schema: joint angles, action class, quality score 1–10. Cost: $0.50–2/clip | On demand |
