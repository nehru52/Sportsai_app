// Consumer App First · AI Engine Later · Zero Spend

Build the addictive app
now. Plug in the
engine when it's ready.

Your co-founder owns the AI engine. You build the consumer product — every screen, every addictive loop, every feature — with mock analysis data first. When the real engine ships, swapping it in is one file change.

Kiro — spec-driven features

All backend routes, database schema, gamification logic, API integration, complex multi-file features. Kiro's spec → design → tasks workflow means nothing is built on assumptions.

Trae — visual screens

Every React Native screen built from the design mockups. Drag screenshots into Trae Builder, get matching code. All UI/UX, animations, and interactions. Zero backend knowledge required.

// 00 — Core strategy

Why consumer app first is the right call

The AI engine is being built separately. You build everything the user touches. When the engine ships, it connects to an API you've already built and tested. The consumer app is the harder problem anyway.

The key insight: The real AI analysis — pose extraction, rules engine, Claude feedback — all returns a standard JSON object: { score, phase_scores, errors, feedback_text }. You build your entire app against this interface using mock data. When the real engine ships, you change one environment variable. Every screen, animation, gamification event, and notification works immediately.
What Kiro does that changes everything for you: Kiro's spec-driven development turns your natural language into EARS-notation requirements, then architecture, then implementation tasks with acceptance criteria. You type "build the XP system" — Kiro writes requirements.md, design.md, and tasks.md before touching code. You review and type "LGTM" to proceed. No more AI making assumptions you discover in code review.

// 01 — Build phases

3 phases — exactly what ships when

Phase 1 · Weeks 1–6

Core consumer app

Onboarding, home screen, record + upload, analysis screen with mock data, full gamification (XP, belts, streaks, challenges, skill tree). Everything works. Analysis uses mock scores that look and feel real.

Phase 2 · Weeks 7–10

Addiction layer

1v1 Arena, AI coach chat (using Claude API directly — this IS real from day 1), social leaderboards, anime cinematics with DomoAI, avatar system, rivalry cards, push notifications, all retention mechanics.

Phase 3 · On engine ready

Real engine plug-in

Set EXPO_PUBLIC_USE_REAL_ENGINE=true. Real pose analysis replaces mock. Skeleton overlay works with real data. Coach marketplace opens. Everything else already built — zero app rebuilding needed.

The AI coach chat is real from Day 1. It uses Claude API directly — not the pose engine. The coach reads the user's history (stored in Supabase from their sessions) and gives real coaching advice. This is your most powerful feature and costs nothing to build. It works with mock session data just as well as real data.

// 03 — Agent hooks

Kiro agent hooks — set up once, run forever

These hooks run automatically in the background as you build. You set each one up once by pasting the prompt into Kiro's hooks panel. They save hours of manual work across the entire project.

Trigger: any *.tsx file saved in app/ or src/components/
Hook 1 — Auto-generate test file
When I save any React Native screen or component, automatically create or update its corresponding test file in __tests__/ using React Native Testing Library. Tests should cover: renders without crashing, loading state shows, error state shows, primary CTA is present and tappable. Follow the existing test patterns in the project.
Trigger: src/types/index.ts saved
Hook 2 — Sync API types to backend
When I update the TypeScript types in src/types/index.ts, check if any changed interface breaks the corresponding Supabase schema or API route response shapes. If there's a mismatch, create a GitHub issue comment listing what needs updating in sportsai-api. Never auto-change the API — just flag the discrepancy.
Trigger: any file in src/services/ saved
Hook 3 — Security scan
When any service file is saved, scan for: hardcoded API keys or secrets, direct fetch() calls (should use src/lib/api.ts), missing error handling on async calls, any `any` TypeScript type. Output a warning comment at the top of the file if any issue is found. Add problematic files to .gitignore if they contain secrets.
Trigger: any new screen created in app/
Hook 4 — Design system check
When a new screen file is created, check that it: imports colors from src/theme/colors (not hardcoded hex), uses StyleSheet.create (no inline styles), imports fonts from src/theme/typography, uses react-native-reanimated for animations (not Animated from RN), uses Expo Router navigation (not React Navigation). Add a TODO comment for any violation found.
Trigger: manual trigger (run before any commit)
Hook 5 — Pre-commit quality check
Run TypeScript type checking (tsc --noEmit), check for any TODO comments added by other hooks (surface them for review), verify no env vars are hardcoded, check that every new screen has a corresponding entry in SCREENS.md. Output a summary of any issues found. Do not block the commit — just report.
To set up each hook: In Kiro → left panel → Hooks section → New Hook → paste the description above as the instructions → set the trigger event. Kiro generates an optimized system prompt from your description automatically.
